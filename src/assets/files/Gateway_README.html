GATEWAY

Project Overview

This service acts as the API Gateway and sole entry point in a microservice ecosystem. Built on Spring Cloud Gateway (Reactive), its primary responsibilities include:

  • Centralized Edge Routing & Load Balancing.
  • Frontline Security Enforcement (OAuth2 JWT Validation, CORS, and CSRF mitigation).
  • Tenant Context Extraction and Request Enrichment via custom filter chains.

Technology Stack & Compatibility

  • Java Version: 17
  • Spring Boot Version: 3.5.5
  • Build Tool: Maven
  • Core Dependency: spring-cloud-starter-gateway

Key Features

Security & Token Validation

As the edge proxy, this application is configured as an OAuth2 Resource Server validating stateless JSON Web Tokens (JWTs).

  • Token Verification: Done automatically via the JWK Set URI configured under the identity provider properties.
  • Reactive Adapter: Uses ReactiveJwtAuthenticationConverterAdapter to map JWT claims into the reactive security context.

Custom Gateway Filter Chain

Routing logic goes beyond basic path forwarding. We employ a custom MyGatewayFilter (implemented via GatewayFilter) applied explicitly inside a RouteLocator bean.

The gateway filter sequence enforces the following pipeline:

  • Header Validation & Guardrails: Validates that all mandatory headers are present on the incoming request (e.g., verifying the presence of the required X-XSRF-TOKEN, tenant identifiers, or expected metadata). If any required header is missing, the request is immediately rejected with an appropriate HTTP error status (e.g., 400 Bad Request or 403 Forbidden) without forwarding.
  • Authentication Double-Check: Validates the presence and validity of the security context.
  • Authorization: Checks user roles/scopes against the target path.
  • Tenant Extraction: Resolves multi-tenant variables (TenantInfo), injecting them into the downstream request headers.
  • Downstream Routing: Forwards the mutated request to the target destination.

Prerequisites & Configuration

Identity Provider (IdP) Integration

The gateway requires a running JWKS endpoint to decode and verify JWT signatures. Ensure application properties mirror the following structure:

spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          jwk-set-uri: https://auth-server.com/auth/.well-known/jwks.json

CORS & CSRF Mitigation

Because this is a web-facing boundary, strict cross-origin rules apply:

  • CORS: Handled via a global CorsWebFilter bean to manage cross-origin traffic explicitly.
  • CSRF: Configured using a cookie-backed repository optimized for Single Page Applications (SPAs).

Architectural Implementation Details

Security Configuration

The core security configuration sets up a stateless SecurityWebFilterChain. A crucial component here is the custom CsrfFilter (implementing standard WebFlux WebFilter) used to initialize and subvert deferred token issues by triggering the token on the first GET request.

// Conceptualized CSRF & Filter configuration within Configuration file
.csrf(csrf -> csrf
    .csrfTokenRepository(cookieServerCsrfTokenRepository()) // Bean method name
    .csrfTokenRequestHandler(new ServerCsrfTokenRequestAttributeHandler())
)

Filter Ordering: The custom CsrfFilter is explicitly configured to execute immediately after SecurityWebFiltersOrder.CSRF to ensure it subscribes to and forces the generation of the token cookie.

Dynamic Route Routing

Routes are explicitly managed as Java beans rather than raw configuration files to cleanly embed the custom filter logic.

@Bean
public RouteLocator customRouteLocator(RouteLocatorBuilder builder, MyGatewayFilter gatewayFilter) {
    return builder.routes()
            .route("portfolio-service", r -> r.path("/portfolio/**")
                    .filters(f -> f.filter(gatewayFilter))
                    .uri("https://my-portfolio.com")
            )
            .build();
}

Decision Log (Architecture Decision Records)

ADR 1: WebFlux over Spring MVC

  • Context: Gateway apps experience high-concurrency I/O waiting on downstream microservices.
  • Decision: Use spring-cloud-starter-gateway (built on Netty/WebFlux) instead of the old Servlet-bound blocking gateway versions.
  • Consequence: BlockHound or reactive paradigms must be adhered to. Traditional ThreadLocal cannot be used safely for multi-tenancy context propagation.

ADR 2: Java Route Definitions over YAML

  • Context: Routes require deep integration with a highly specific, imperative custom filter (MyGatewayFilter).
  • Decision: Define paths using the RouteLocator fluent Java API.
  • Consequence: Changing routes requires a application rebuild/redeploy, but guarantees strict type-safety and robust filter ordering.

Troubleshooting

Tracking Routing Decisions

If traffic is returning 404 Not Found or hitting the incorrect microservice, elevate the logging thresholds in local configuration profile to expose the gateway's internal evaluation pipeline:

logging:
  level:
    org.springframework.cloud.gateway: TRACE
    org.springframework.security: DEBUG

Missing CSRF Cookie on Initial Load

If front-end applications fail because of a missing XSRF-TOKEN cookie, verify that the client application sends an initial metadata GET request. The custom CsrfFilter will not flush the cookie to the client until a subscription is explicitly triggered on that initial connection loop.

Future Enhancements

  • Migration of the SecurityWebFilterChain, CorsWebFilter, and generalized security definitions out of this local project and directly into the internal shared library (shared-service). This will ensure that if additional gateway instances are spun up for other sub-domains, security protocols remain identical.
